# 핵심 주장

RAG는 질문할 때마다 원문 조각을 재발견할 뿐 지식이 **축적되지 않는다**.
대신 LLM이 원자료와 사용자 사이에 **영속적이고 복리로 쌓이는 위키**를
만들고 유지보수한다. 소스를 추가하면 색인만 하는 게 아니라 읽고, 추출하고,
기존 페이지들에 통합한다 — 상호 참조는 이미 걸려 있고, 모순은 이미
표시되어 있고, 종합은 이미 모든 소스를 반영한 상태가 된다.

역할 분담: 사람은 소싱·탐색·질문을, LLM은 요약·상호참조·파일링·부기를
맡는다. "Obsidian은 IDE, LLM은 프로그래머, 위키는 코드베이스."

# 아키텍처 — 3계층

1. **Raw sources**: 불변의 원자료. LLM은 읽기만 한다.
2. **Wiki**: LLM이 전적으로 소유하는 마크다운 디렉터리.
3. **Schema**: 위키 구조·규약·워크플로를 정의하는 설정 문서(CLAUDE.md).
   LLM을 "규율 있는 위키 관리자"로 만드는 핵심이며 사용자와 공진화한다.

# 운영 — 3가지 오퍼레이션

- **Ingest**: 소스 읽기 → 논의 → 요약 페이지 → 관련 페이지들 갱신(모순
  표시 포함) → index 갱신 → log 기록. 소스 하나가 10~15페이지를 건드릴 수 있다.
- **Query**: index 먼저 → 드릴다운 → 인용과 함께 답변. **좋은 답변은
  위키에 되기록**해 탐색도 복리로 쌓는다.
- **Lint**: 모순, 낡은 주장, 고아 페이지, 누락 개념, 빠진 상호참조 점검.

# 이 위키에의 적용

이 문서의 패턴을 OKF v0.1 형식과 결합해 적용했다 — LLM Wiki가
**프로세스**(ingest/query/lint, 스키마 공진화)를, OKF가 **형식**(frontmatter,
type, index/log 구조, 적합성 기준)을 담당한다. 구체 규칙은
[Wiki Schema](../CLAUDE.md) 참고. 중간 규모까지는 index 기반 탐색으로
충분하며(~100소스), 커지면 검색 도구(qmd 등) 도입을 검토한다.
